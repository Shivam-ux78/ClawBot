ClawBot

An AI + human hybrid system that finds Instagram creators, sends cold outreach DMs through a headless browser, negotiates deal terms with GPT-4o, and routes every decision point to a human operator over Telegram.

Node.js / Express PostgreSQL Redis + BullMQ Puppeteer OpenAI gpt-4o Telegram Bot API Chrome Extension (MV3)

01 · Flow

One creator, five stages

Everything the system does routes a creator through the same five stages. Human approval gates sit at stage 2 and stage 5 — the bot never sends money-committing messages unsupervised unless Auto Mode is explicitly turned on.

01 Discover Cron scrapes IG location + category hashtags, scores candidates, inserts into creators
02 Approve Telegram card → operator taps Approve/Reject (skipped in Auto Mode)
03 Outreach BullMQ job delays 60–90s, Puppeteer follows + DMs from a real session
04 Negotiate Creator reply → GPT-4o drafts counter or reads a quoted price against budget
05 Close Deal card → operator accepts/rejects → confirmation or decline DM queued
Operator (Telegram) ClawBot server OpenAI gpt-4o Instagram (via cookies)

05 · Subsystem

Discovery engine — location-first, then category

src/instagram/discover.js runs on a cron (default every 6h) and again automatically 30 minutes after any run that finds nothing. It deliberately searches geography before niche:

StepWhat happens
1Open a US location hashtag (country → state → city, e.g. #losangeles) and collect recent post/reel links.
2For each post, read the author and the post's own hashtags.
3Keep the post only if its hashtags intersect the category set (couple/relationship/etc) — skippable via /StopCategoryFilter.
4–5Visit the author's profile: reject if follower count is out of range, account is private, or "About this account" names a clearly non-US country.
6GPT-4o-mini classifies the profile into one category from a fixed list; must land in the keep-list.
7Compute a 0–100 confidence score (below); reject under the configured threshold.
8On a qualifying creator, sample their followers/following — creators cluster geographically — and run steps 4–7 on those too.

Confidence score

+40
Location = US
+20 / −10
Bio keyword score
+30
Post themes match
+10
GPT category kept

Default threshold is 80%. Every rejection is logged with its score and reason so a scan can be tuned after the fact.

Operator levers

/range (follower bounds) · /settarget (location + category) · /StopCategoryFilter / /StartCategoryFilter · /Auto / /Manual · /stop <hours> / /startscan — all writable live from Telegram, persisted to settings.

06 · Subsystem

AI negotiation

src/ai/negotiate.js wraps GPT-4o behind one system prompt: warm, concise, never reveals budget ceilings. Budget logic is a pure function over three configured numbers.

Creator's quoted priceDecisionOutcome
> MAX_BUDGETcounterGPT drafts a counter-offer anchored near TARGET_BUDGET
MIN ≤ price ≤ MAXpropose_dealDeal card sent to Telegram for accept/reject
< MIN_BUDGETacceptStill routed through a deal card — "too good to auto-accept unseen"

Price extraction is regex-based (extractPrice()), not model-based — it matches $200, 200 USD, 200 dollars, etc. Reply generation, counter-offers, and deal confirmations are three separate prompted calls, each capped at 150–300 tokens to keep messages DM-length.

07 · Subsystem

Queue & rate limiting

Every outbound DM — outreach, counter, decline, confirmation — goes through one BullMQ queue (dm-queue) so the worker process is the only thing that ever touches Puppeteer.

Randomized delay
60–90s by default (DM_DELAY_MIN_SEC / _MAX_SEC), overridable per job
Daily cap
DM_DAILY_LIMIT (default 40) — worker counts dm_log rows since midnight and skips the job rather than erroring
Retry policy
3 attempts, exponential backoff from 5s; jobs are removed on both completion and final failure
Concurrency
1 — DMs are sent strictly one at a time through a single hidden browser
Single point of failure

The worker is a separate, always-on Node process (npm run worker locally, or the clawbot-worker service on Render). If it isn't running, creators move through approval but no DM ever actually leaves — this is the first thing to check when outreach "isn't sending."

09 · Subsystem

Instagram delivery — cookies over API

There's no official Instagram DM-send API for this use case, so outreach and discovery both drive a real, logged-in session through headless Chrome.

Auth source
Exported browser cookies, read from Redis key ig_cookies first, falling back to www.instagram.com.cookies.json on disk
Cookie freshness
The Chrome extension re-pushes cookies to POST /api/cookies/update every 30 min while the operator's browser is open and logged into the matching account
Send sequence
Navigate to profile → click Follow → open Message composer → dismiss notification prompt → type with per-keystroke delay → Enter
Stub mode
INSTAGRAM_STUB_MODE=true logs the DM to console instead of launching a browser — the default for local dev
Fragility

Both DM-sending and discovery scrape Instagram's live DOM (button text, dialog structure, meta-description follower counts). Any Instagram front-end change can silently break follow/message detection or profile parsing until selectors are updated.

12 · Operations

Deploying on Render

The repo ships a Dockerfile and a render.yaml blueprint. Docker is used specifically because Puppeteer needs a real Chromium binary and its shared libraries — a plain Node buildpack doesn't provide those reliably, so the image installs chromium via apt-get and points Puppeteer at it via PUPPETEER_EXECUTABLE_PATH.

clawbot-web

Docker service

Runs node src/index.js — Express API, Telegram bot (polling), discovery cron. Health check on /health.

clawbot-worker

Docker service

Same image, dockerCommand overridden to node src/queues/worker.js. Telegram bot runs in send-only mode here to avoid double-polling.

What stays external

Postgres and Redis are not provisioned by the blueprint — DATABASE_URL / REDIS_URL are declared as secrets in the clawbot-secrets env var group and filled in against Supabase/Upstash (or Render's own Postgres/Key Value — only the connection string changes).

Cookies on an ephemeral filesystem

Render wipes the container filesystem on every redeploy, so the local www.instagram.com.cookies.json fallback is dev-only by design. In production, the Chrome Extension pushes fresh cookies to Redis (ig_cookies key) every 30 minutes via POST /api/cookies/update — this was already the primary path even before Render entered the picture; the file was only ever a local-dev convenience.

Do not autoscale clawbot-web

node-cron (discovery scheduling) and the Telegram bot's long-polling both assume exactly one running process. A second instance means two discovery scans firing at once and a Telegram 409 Conflict as both pollers grab the same update. Keep numInstances: 1.

13 · Read before operating

Notes for whoever runs this

Hardcoded admin password

src/telegram/handlers.js gates /AddID and /RemoveID behind the literal string "Admin123@" checked in plaintext — anyone who can message the bot and knows or guesses this can add themselves as an approval recipient. Worth moving to an env var at minimum.

Real outreach is genuine automation on your IG account

With INSTAGRAM_STUB_MODE=false, the bot follows and messages real accounts using your logged-in session cookies. This is exactly the kind of automated, non-official-API behavior Instagram's terms prohibit and can flag or restrict the account — treat rate limits and delays as a floor, not a guarantee of safety.

Selector fragility

Discovery and DM-sending both depend on scraping Instagram's current DOM structure and English-language button text ("Follow", "Message", "Not Now"). Expect breakage after Instagram redesigns without warning.

Worker is a separate process

Nothing is actually sent unless npm run worker (or the worker Procfile process, on Railway/Heroku-style hosts) is running alongside the main server.
