Approach & requirements

Search → Re-match → Send → Notify, on two platforms

You want one loop that already works for Instagram — search under a first set of conditions, re-check candidates against a second, tighter condition, and only then send a DM and tell you on Telegram — extended to LinkedIn, with a web app to run and watch it. Below is the concrete approach, what's reused, what's new, and what to build in what order.

Instagram — extend existing engine LinkedIn — new connector Web dashboard — new Telegram — stays as-is

01 · The core idea

This is already how ClawBot's Instagram engine works

Look at src/instagram/discover.js: it doesn't search once and act — it runs a two-stage filter. That pattern is exactly what you described, and it's the thing to generalize across platforms rather than rebuild per-platform.

STAGE 1Search conditionBroad discovery filter — e.g. location hashtag on Instagram, or search keywords + industry + title on LinkedIn
STAGE 2Re-match conditionNarrower qualifying filter against each candidate — e.g. category hashtag match, follower range, bio keywords, AI classification, confidence score
DECISIONMatch?Score ≥ threshold → proceed. Below → discard and log why, same as today's confidence gate
ACTSend DMQueued, rate-limited, human-paced send through the platform connector
NOTIFYTell the operatorTelegram message (and a live row in the web dashboard) — same notify() call used today

The only thing that changes per platform is what Stage 1 and Stage 2 look like and how the DM is physically sent. Everything else — the queue, the rate limiting, the Telegram notifications, the deal/negotiation logic — is platform-agnostic already and should not be duplicated.

02 · Architecture

One connector interface, two implementations

Introduce a small contract every platform must implement. Instagram's existing code becomes the first implementation of it (a refactor, not a rewrite); LinkedIn becomes the second (new code).

interface PlatformConnector {
  search(conditions: SearchConditions): Promise<Candidate[]>;
  evaluate(candidate: Candidate, conditions: MatchConditions): Promise<MatchResult>;
  sendMessage(target: Candidate, message: string): Promise<SendResult>;
  getSessionHealth(): Promise<SessionHealth>;
}
connectors/instagram/

Existing engine, wrapped

discover.js → search() + evaluate(). client.js → sendMessage(). Behavior unchanged, just exposed through the shared shape.

connectors/linkedin/

New

Puppeteer-driven, cookie-authenticated, same shape. Search runs LinkedIn people-search; evaluate reads the profile page; sendMessage sends a connection-request note or a direct message if already connected.

core/matchingEngine.js

Shared

Generalizes today's computeConfidence() into a config-driven scorer: a list of weighted rules, a threshold, applied identically regardless of which connector produced the candidate.

core/prospectService.js

Shared (renamed creatorService)

Platform-tagged version of today's creator lifecycle — add / approve / reject / message-log, generic across Instagram and LinkedIn rows.

queues/outreachQueue.js

Shared (renamed dmQueue)

Same BullMQ pattern, one queue per platform (separate rate limits — LinkedIn's safe daily ceiling is much lower than Instagram's) but identical job-processing shape.

03 · Why LinkedIn needs its own web app

Telegram cards stop being enough

Instagram's operator surface is entirely Telegram because the decisions are simple: approve, reject, accept a price. LinkedIn outreach has more moving parts that don't fit an inline-keyboard card, which is the real reason to build the dashboard now rather than force everything through chat:

Stays in Telegram

  • Approve / reject a queued candidate — still a 2-button decision
  • Deal accept / reject — unchanged
  • Alerts — new reply, account warning, daily cap hit
  • Quick pipeline glance — /list, /status equivalents

Moves to the web dashboard

  • Building a condition set — LinkedIn search filters (title, industry, location, company size, keyword) don't fit a chat command
  • Reviewing a candidate list before sending — a table you can sort/filter/bulk-approve, not one card at a time
  • Editing message templates per campaign, with variables
  • Account health — connection-request acceptance rate, weekly invite count used, warning banners from LinkedIn — needs a chart, not a chat line
  • Multi-account management once you run more than one LinkedIn seat
Design implication

The web app is a campaign & review console, not a replacement control channel. Telegram remains the fast path for yes/no decisions and alerts on both platforms; the dashboard is where you configure conditions, watch pipelines, and manage LinkedIn accounts at a level of detail chat can't express.

04 · End-to-end

What actually happens on a LinkedIn run

StepDetail
1Operator defines a campaign in the web dashboard: search keywords, title/industry filters, location, connection-degree limit, daily send cap.
2Scheduler (same node-cron pattern as today) triggers the LinkedIn connector's search() — runs LinkedIn's people-search URL through Puppeteer with the operator's session cookies.
3Each result goes through evaluate(): open profile, read headline/about/experience, score against Stage 2 conditions (role keywords, seniority, company size) — same weighted-confidence shape as Instagram.
4Matches ≥ threshold land in the shared prospects table, platform = linkedin. Telegram gets an approval card; the dashboard's review queue gets a row.
5On approval, a job is queued: send a connection request with a personalized note (first touch — LinkedIn requires this before free-form messaging in most cases), rate-limited and delayed like Instagram's outreach queue.
6When the request is accepted (polled periodically), the real outreach/negotiation message queues automatically and Telegram is notified: "@name accepted — DM sent."
7Replies land back through the same webhook/notify pattern used today — log to conversations, notify Telegram, surface in the dashboard thread view.

05 · Data model

Generalize, don't fork

Add a platform column rather than a parallel LinkedIn schema — the shape of "prospect moving through a pipeline" is identical, only the metadata attached to a row differs.

ALTER TABLE creators RENAME TO prospects;
ALTER TABLE prospects ADD COLUMN platform VARCHAR(20) NOT NULL DEFAULT 'instagram';
ALTER TABLE prospects ADD COLUMN platform_meta JSONB;  -- title, company, connection_degree, etc.

CREATE TABLE campaigns (
  id             SERIAL PRIMARY KEY,
  platform       VARCHAR(20) NOT NULL,
  name           VARCHAR(255) NOT NULL,
  search_conditions   JSONB NOT NULL,   -- Stage 1
  match_conditions    JSONB NOT NULL,   -- Stage 2 + weights + threshold
  daily_send_cap INTEGER NOT NULL DEFAULT 20,
  active         BOOLEAN NOT NULL DEFAULT true,
  created_at     TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE TABLE platform_accounts (
  id           SERIAL PRIMARY KEY,
  platform     VARCHAR(20) NOT NULL,
  label        VARCHAR(255) NOT NULL,     -- "main LinkedIn seat"
  cookies_key  VARCHAR(255) NOT NULL,     -- Redis key, mirrors ig_cookies pattern
  daily_used   INTEGER NOT NULL DEFAULT 0,
  status       VARCHAR(20) NOT NULL DEFAULT 'healthy', -- healthy / warned / restricted
  updated_at   TIMESTAMP NOT NULL DEFAULT NOW()
);

conversations, deals, and dm_log need no schema change beyond following the rename — they already key off creator_id / prospect_id and don't care which platform it came from.

06 · Requirements

Functional

IDRequirement
FR-1Operator can define a campaign per platform with a Stage-1 search condition and a Stage-2 re-match condition, each independently editable.
FR-2Every candidate is scored by the shared matching engine; only candidates at or above the campaign's threshold enter the pipeline.
FR-3New qualifying candidates trigger a Telegram approval card and a dashboard queue row simultaneously.
FR-4Approval queues a platform-appropriate first-touch send (DM on Instagram; connection request + note on LinkedIn) via the rate-limited outreach queue.
FR-5LinkedIn connection acceptance is polled and automatically advances the prospect to "message sent" without a second manual approval, unless Auto Mode is off.
FR-6Every send and every reply is logged to conversations and mirrored to Telegram, regardless of platform.
FR-7Dashboard shows, per platform account: daily sends used vs. cap, and a status flag (healthy / warned / restricted).
FR-8Dashboard authentication is required — this is an operator console, not a public site.

Non-functional

Safety-first pacing
LinkedIn's safe daily action ceiling is far lower than Instagram's — connection requests are commonly kept under ~20–25/day per account, vs. Instagram's 40 DM/day default. This must be a per-platform config, not a shared constant.
Session resilience
Both connectors depend on live browser cookies; the dashboard must surface "cookies stale" the same way getProfileInfo() already warns today, before a run silently fails.
Extensibility
A third platform should mean writing one new connector file, not touching the queue, matching engine, or Telegram layer.
Auditability
Every rejected candidate keeps its score and the reason it failed Stage 2 — mirrors today's discovery logging, needed to tune thresholds later.

07 · LinkedIn specifics

What's actually different from Instagram

AspectInstagram (today)LinkedIn (new)
AuthExported cookies, pushed by the Chrome extensionSame pattern — extension already generalizes to any *.instagram.com-style host; add linkedin.com cookie capture
Search surfaceHashtag explore pagesLinkedIn people-search URL with query params (keywords, title, location, industry) — no Sales Navigator required for the base case, but Sales Navigator filters are far richer if the account has it
First contactDirect DM immediatelyUsually requires a connection request with an optional short note (≈200 chars) first; free-form messaging is normally gated behind an accepted connection or a paid InMail credit

Detection surfaceDOM scraping, human-paced typingSame techniques apply, plus LinkedIn is more aggressive about flagging automated browsing patterns and known-headless fingerprints — expect to invest more in stealth (viewport/user-agent variation, longer randomized delays) than Instagram needed

09 · Cloud target

Everything above assumes Render, not a bare VM

The current Instagram-only system already runs on Render as a Dockerfile + render.yaml blueprint (see docs/architecture.html). Every new piece proposed here needs to slot into that same shape rather than introduce a second deployment story.

PieceRender shape
LinkedIn connectorRuns inside the existing clawbot-worker service — same Docker image, same Chromium install, one more BullMQ queue lane (linkedin-outreach) alongside the Instagram one, each with its own daily cap.
Discovery scheduling per platformStays in clawbot-web's node-cron — same singleton-instance constraint applies to whichever platform's scan is running.
Web dashboard (React)Two options: (a) build it as static assets and serve them from the existing Express app (clawbot-web) behind the same domain, zero new services; or (b) a separate Render Static Site service calling the Express API as its backend. Start with (a) — one service to operate is simpler until the dashboard has its own release cadence.
Dashboard authA session/JWT secret becomes a new entry in the clawbot-secrets env var group — same mechanism already used for EXTENSION_SECRET_KEY.
LinkedIn cookiesSame Redis-backed pattern as Instagram's ig_cookies — a li_cookies key, pushed by extending the existing Chrome extension to also capture linkedin.com cookies. No new infrastructure.
Account health pollingA lightweight scheduled check (reuse node-cron in clawbot-web) that updates platform_accounts.status — surfaced in both Telegram alerts and the dashboard, not a separate service.
Still one web instance

Adding a dashboard doesn't relax the earlier constraint — clawbot-web stays at numInstances: 1 regardless of how much traffic the dashboard gets, because the cron and Telegram polling living in that same process are still singletons. If dashboard load ever becomes a real concern, split it into its own stateless service that only talks to Postgres/Redis — never let it share a container with the cron/polling process.

10 · Read before building

Constraints that shape the design above

LinkedIn enforcement is stricter than Instagram's

LinkedIn actively litigates and technically fights scraping/automation more aggressively than most platforms, and account restrictions can be faster and harder to reverse. Everything above assumes conservative daily caps and a willingness to run manual-approval-only (no LinkedIn equivalent of Auto Mode) until account health data justifies otherwise.

Reuse over rewrite

The single biggest risk to this project is building LinkedIn as a parallel system instead of a second connector. If discover.js's two-stage scoring, the BullMQ queue shape, and notify() aren't extracted into shared modules first (Phase 1), every platform added after LinkedIn doubles the maintenance surface again.

Dashboard auth is not optional

Unlike Telegram (implicitly scoped to registered chat IDs), a web dashboard is reachable by URL. It needs real authentication before it ships, not after — see the hardcoded-password note already flagged in the current architecture doc for what happens when this gets deferred.
